# Models-Compare-For-Churn-Prediction

# Цель работы - провести сравнительный анализ классических алгоритмов машинного обучения для прогнозирования оттока клиентов.

## Описание датасета:

Набор данных Online Retail II содержит все транзакции, совершенные зарегистрированным в Великобритании онлайн-магазином розничной торговли, не связанным с магазинами, в период с 01.12.2009 по 09.12.2010.Компания в основном занимается продажей уникальных сувениров на все случаи жизни. Многие клиенты компании являются оптовыми торговцами.

Признаки:

InvoiceNo - Номер счета-фактуры. Номинальный. 6-значный цельный номер, однозначно присваиваемый каждой транзакции. Если этот код начинается с буквы "c", это означает отмену транзакции.

StockCode - код продукта (номенклатуры). Номинальный. 5-значное целое число, однозначно присваиваемое каждому отдельному продукту.

Description - Название продукта (позиции). Номинальный.

Quantity - количество каждого продукта (номенклатуры) в каждой транзакции. Числовой.

InvoiceDate - дата и время выставления счета. Числовой. День и время, когда была произведена транзакция.

UnitPrice - цена за единицу товара. Числовой. Цена за единицу товара в фунтах стерлингов (€£).

CustomerID - номер клиента. Номинальный. 5-значный цельный номер, уникально присваиваемый каждому клиенту.

Country - Название страны. Номинальный. Название страны, в которой проживает клиент.

## EDA анализ транзакционного датасета:

Были построены графики Box-plot, Plot. Было обнаружено большое количество экстремальных значений для числовых признаков Quantity и UnitPrice, которые были обусловлены финансовыми операциями (банковские сборы и т.п.) для UnitPrice и товарами содержащих большое количество штучных товаров в пачках (например салфетки, ручки и т.д.) для Quantity.

## Feature Engineering:

Для анализа необходимо было перевести датасет транзакционный в датасет клиентский. Для этого изначальный датасет был сгруппирован по Customer ID и для сгруппированного датасета были созданы различные признаки, описывающие поведение клиента.

Целевая переменная Churn была построена по следующему принципу:

Churn задавалась на основе временного окна наблюдения и прогнозного горизонта. Клиент относился к классу оттока в случае, если после контрольной даты в течение выбранного горизонта не совершал новых покупок. Подобная реализация нужна для предотвращения утечки данных.

Итоговый датасет имеет следующий вид:
Customer ID: уникальный идентификатор клиента, каждая запись в датасете является записью со статистикой покупок отдельного клиента;

TotalPrice - суммарная стоимость покупок клиента;  

Entry_Counts - число записей о покупках;  

TotalQuantity - общее количество приобретенных товаров;  

Last_InvoiceDate – дата последней покупки; 

Country - страна клиента; 

Return_Count, Return Quantity – количество записей возвратов и количество возвращенных товаров;  
 
Frequency - число заказов; 

Unique_Products - число уникальных товаров;  

Avg_Items_per_Order - среднее количество товаров на заказ;  

Avg_Price_per_Order - средняя стоимость заказа;  

Max_Inactivity_Days - максимальный период неактивности;  

Had_Long_Gap, Is_One_Time_Byuer, Is_Few_Purchase - признаки, характеризующие разовый или редкий характер покупок;  

Recency - показатель давности последней покупки;  

Avg_Days_Between_Orders - средний интервал между заказами;  

Customer_Stage - категориальные признак, описывающие стадию клиента, one_time – клиенты с одним заказов, loyal – клиенты с большим числом заказов и малым интервалом между заказами, active – регулярные клиенты с меньшей активностью чем loyal, sleeping – клиенты с выраженным снижением активности; 

Purchase_Band - показывает категорию клиента, если число закзаов = 1 - 'one', меньше 3 - 'few', меньше 10 - 'medium', больше 10 - 'many'.

## Подготовка моделей:

Сравнительный анализ в работе строился на четырех алгоритмах: Logistic Regression, Decision Tree, Random Forest и CatBoost. Выбор именно этих моделей обусловлен необходимостью сопоставить методы разных классов: от простой интерпретируемой линейной модели до современных ансамблевых алгоритмов, ориентированных на сложные табличные данные.

После разбиения выборок были использованы OneHotEncoder для кодирования категориальных признаков и StandartScaler для стандартизации признаков числовых.

## Обучение моделей и оценка результатов:

После подбора гиперпараметров с помощью RandomizedSearchCV модели были обучены. 

Точность моделей оцененная методом score следующая:

Логистическая регрессия - 0.676

Дерево решений - 0.656

Случайный лес - 0.674

Бустинг - 0.673

Далее был проведен анализ влияния дисбаланса классов на метрики F1, precision, recall с помощью Classification Report и матрицы ошибок. 

Логистическая регрессия при отстутствии методов балансировки классов показывает хорошее качество для класс 0(примерно 0.72) и хуже для класса 1(примерно 0.61), с сэмплированием вниз показатели выравниваются класс 0 стал предсказываться хуже (примерно 0.70), а класс 1 лучше (примерно 0.64), лучшая модель будет определяться целлью классификации: если нам важно, чтобы 1 класс предсказывался лучше, то стоит выбрать метод с сэмплированием вниз.

Для Дерева решений ситуация похожа, метрики для модели без сэмплирования почти такие же как и для лог регрессии, похожая ситуация и с моделью с сэмплированием данных вниз, однако модель с сэмплированием вверх показала результаты лучше остальных: для класса 0 метрика f1 - 0.70, для класса 1 - 0.62

Для Случайного леса существует параметр class_weight как раз для решения проблемы балансировки классов, с данным методом класс 0 имеет показатель 0.69, класс 1 - 0.65, для остальных методов эти цифры релевантны, различия +- 1 процент.

Для бустера также есть аналогичный предыдущему алгоритму праметр для балансировки классов, для всех методов балансировки класс 1 имеет метрику равной 0.66, для остальных применро 0.69, для сэмплирования вверх она больше остальных, но лишь на 1-2 процента.
